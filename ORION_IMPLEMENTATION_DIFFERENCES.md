# Orion vs OrionBC: Implementation Differences

This document compares the official Orion implementation (paper/repository) with this OrionBC implementation in more depth. Each row includes: (1) what Orion does, (2) what OrionBC does, and (3) why OrionBC is simplified, constrained by available datasets, or still sufficient for the current project objective.

## Detailed Comparison Table

| Component | Official Orion implementation (paper/repo) | OrionBC implementation (this repo) | Why simplified / dataset limits / why currently sufficient |
|---|---|---|---|
| Problem framing and deployment target | Orion is designed as a robustness-first clinical screening model, especially for early-stage NSCLC detection across heterogeneous suppliers/cohorts. The method and code are oriented toward reducing cohort/batch artifacts and improving external validity. | OrionBC is framed as an Orion-inspired research implementation for breast tissue/circulating tasks plus a large serum pan-cancer dataset. It is optimized for reproducible local experimentation and teaching-oriented interpretability. | The project goal is benchmarking and understanding, not immediate clinical deployment. This justifies a simpler baseline while still preserving the core generative + discriminative structure. |
| Expected data dictionary and modality assumptions | Official training expects explicit `oncrna_ar` (signal), `smrnamat` (small RNA for content normalization), `onehot_ar` labels, and `batch_list` (for anchor/triplet domain structure). | OrionBC uses a single matrix per dataset, then derives two feature subsets: `signal_features` and `ref_features`. Labels are binary or multiclass tensors built from local metadata/rules. | Public GEO datasets used here do not consistently provide Orion-style paired modality and standardized batch/study identifiers across all tasks. Splitting one matrix into signal/reference is a practical surrogate that retains the dual-encoder concept. |
| Dataset curation and metadata harmonization | Orion’s release assumes a curated pipeline and metadata model (cohort, supplier, etc.), with datasets hosted separately (Zenodo) and pre-aligned to training code expectations. | OrionBC loads raw local files (`GSE270497`, `GSE197020`, `GSE211692`) and normalizes schema in `data_utils.py` (feature columns, sample IDs, disease labels, stage labels). | This repo must handle heterogeneous public input formats directly. The extra local relabeling logic is necessary because a shared canonical schema is not available for all datasets. |
| Label generation and class semantics | Orion usually receives pre-defined class labels (`onehot_ar`) and can leverage cohort metadata for robust supervision and analysis stratification. | Tissue labels are inferred by sample-ID pattern; circulating labels are hard-mapped from known sample sets; pan-cancer stage-1 and stage-2 labels are derived from text disease states. | This is a dataset-availability constraint. While less elegant than explicit curation, it is transparent and deterministic, which is sufficient for reproducible experiments in this repository. |
| Feature selection philosophy | Orion examples use odds-ratio + p-value filters and optionally iterative XGBoost refinement, emphasizing biologically plausible and robustness-aware selection. | OrionBC uses fast univariate statistical ranking (`mannwhitneyu`, `ttest`, optional DESeq2) and fixed top-k dimensions (`N_SIGNAL_FEATURES=32`, `N_REFERENCE_FEATURES=64`). | Simpler feature selection reduces runtime and dependency complexity and works well for same-distribution splits. The tradeoff is weaker domain robustness than Orion’s richer filtering strategy. |
| Reference/library feature modeling | Orion learns library/content effects from a dedicated small-RNA matrix and treats this channel as a first-class signal for normalization and classifier robustness. | OrionBC computes reference features by low-CV abundant features from the same matrix and feeds them to a separate encoder. A per-sample library size is computed but not consumed in model forward/training. | Public datasets often lack harmonized paired modality. The proxy reference channel is sufficient to study architecture behavior, but it does not fully replicate Orion’s explicit RNA-content modeling path. |
| Encoder architecture details | Official `VariationalEncoder` is configurable (`hidden_layer_num`, `hidden_dim`, dropout, optional batch/layer norm) and clamps log-variance for numerical stability (`_MIN_LOGVAR=-6`, `_MAX_LOGVAR=6`). | OrionBC hard-codes compact encoders: signal path `Linear(x_dim->128)->LayerNorm->ReLU->(mu,logvar)` and reference path `Linear(r_dim->32)->LayerNorm->ReLU->(mu,logvar)`. | Hard-coding reduces tuning burden and supports small datasets. It is sufficient for proof-of-concept but less flexible than Orion for cross-cohort hyperparameter optimization. |
| Classifier coupling with library scalar | Orion supports explicit scalar injection into classifier input (`concat` or `multiply`) so the classifier can model content effects jointly with latent representation. | OrionBC classifier consumes only `z` latent (`z_dim->16->dropout->out`) and does not inject `l` at classification time. | This branch was omitted to keep implementation minimal and easy to debug. Sufficient for baseline discrimination, but likely weaker under distribution shifts where content normalization matters. |
| Decoder and dispersion parameterization | Orion decoder outputs `px_scale`, `px_dropout`, and uses a global learnable per-feature dispersion parameter (`self.px_r`) exponentiated during reconstruction likelihood. | OrionBC decoder predicts sample-level mean, sample-level dispersion, and sample-level dropout from latent+library hidden states. | OrionBC’s approach is simpler to implement end-to-end in one module, but statistically differs from Orion’s global-dispersion formulation and may behave differently under sparse/noisy regimes. |
| Reconstruction likelihood family | Orion loss supports multiple likelihoods (`zinb`, `nb`, `poisson`, `gaussian`) and uses `scvi` distributions for stable log-likelihood implementations. | OrionBC supports custom ZINB and MSE reconstruction modes. Binary tissue/circulating runs use ZINB; pan-cancer stage runs use MSE. | This lowers dependency friction and complexity. It is sufficient for this project, but narrower than Orion’s configurable probabilistic family. |
| Latent sampling strategy | Orion can train classifier with generative latent sampling (`generative_samples_num`, default often 100), producing `ctpred.generative` and averaging loss across samples. | OrionBC uses a single forward classifier pass without generative classifier sampling. | Removed for simplicity and speed. Adequate for baseline training, but removes one of Orion’s mechanisms for uncertainty-aware and robust decision boundaries. |
| Auxiliary robustness loss (triplet anchors) | Orion includes triplet-margin loss, with anchors sampled by `AnchorFinder` using batch/study/patient structure (positive same class different batch, negative opposite class). | OrionBC omits triplet-margin training entirely and therefore does not use explicit cross-domain anchor constraints. | This is the largest robustness simplification. Without reliable batch/study metadata and anchor sampling, triplet learning is hard to apply consistently. Omission is practical but directly reduces Orion-like domain generalization behavior. |
| Objective composition and weighting | Orion loss is multi-component: reconstruction+KL_l, KL_z, CE, triplet, optional regression reconstruction, all balanced by configurable `loss_scalers`; plus optional L1/L2 penalties and sample-weighted loss. | OrionBC uses compact objective `total = recon + beta*KL + gamma*class` with optional class weights/pos_weight and gradient clipping. | Compact loss is easier to reason about and tune on small datasets. It is sufficient for initial model comparison but lacks Orion’s fine-grained balancing controls for robustness/regularization. |
| Data loader outputs and training context | Orion dataloader passes onctensor/smtensor/batch/labels/sample weights and computes local library priors (`compute_log_lib_params`) used in loss. | OrionBC `ZINBDataset` returns `(x, r, library_size, y)`, but training loops ignore `library_size` (`for x, r, _, y`). | This is a simplification and a gap relative to Orion. The current setup still trains, but misses explicit prior-informed library regularization behavior present in official code. |
| Batch and domain awareness in optimization | Orion trainer/evaluator are domain-aware through `batch_list` and anchor sampling loops (`tm_rounds`), and can enforce robustness through repeated sampled constraints. | OrionBC training is standard minibatch supervised optimization without domain-aware constraints or sampled anchor rounds. | Faster and easier to reproduce; sufficient for within-cohort benchmarking, but expected to underperform under cohort transfer/domain shift. |
| Hyperparameter configurability | Orion exposes broad configuration (`inject_lib`, `use_triplet_loss`, `loss_scalers`, `tm_rounds`, `mini_batch`, `num_lvs`, `n_layers`, `l1/l2`, etc.) via dataclass/config pipeline. | OrionBC uses fewer knobs and mostly fixed architecture/training defaults (`z_dim=32`, `beta=0.01`, `gamma=10`, `lr=1e-3`, fixed hidden widths). | Intentional to reduce complexity and avoid broad grid-search burden. Sufficient for project cadence but less adaptable for production-level tuning. |
| Split protocol and test selection | Orion workflows emphasize strict train/tune separation and robust training logic; deployment analyses are cohort-sensitive. | OrionBC runs many stratified 70/15/15 random seeds, then keeps the model from the best test AUC seed for downstream use. | Multi-seed reporting is useful, but choosing by test AUC is optimistic and biases performance upward. Good for exploration, not for final generalization claims. |
| Pan-cancer task formulation | Orion’s published primary framing is not this exact two-stage pan-cancer decomposition. | OrionBC explicitly builds stage-1 binary cancer-vs-non-cancer and stage-2 13-class cancer-only classifier on `GSE211692`. | Added for project scope and interpretability. Useful for understanding hierarchy of task difficulty, though not a strict replica of Orion’s primary clinical framing. |
| External transfer behavior | Orion architecture/training choices are designed to preserve performance under external shifts (supplier/cohort differences). | OrionBC shows strong in-distribution performance but poor tissue->circulating transfer (reported AUC ~0.38) with low cross-dataset feature overlap. | This outcome is consistent with omitted robustness modules and domain mismatch. It confirms the baseline is learning strong within-domain signals but limited domain-invariant structure. |

## Generalization Gaps and Orion-Aligned Next Steps

The likely causes of weak generalization in OrionBC are additive rather than single-factor: substantial biological/technical domain shift between tissue and circulating cohorts; heavy top-k train-domain feature compression that leaves limited transferable overlap; omission of Orion’s explicit robustness terms (triplet anchor loss and generative classifier sampling); proxy library modeling from the same matrix instead of a dedicated paired modality channel; and optimistic model retention based on best test-seed. To move OrionBC closer to Orion behavior, the highest-yield next steps are: (1) add standardized cohort metadata and pass `batch_list`/`study_name` end-to-end, (2) implement anchor-based triplet-margin loss with tunable `tm_rounds`, (3) add generative latent classifier sampling and corresponding loss path, (4) switch to Orion-style OR/XGBoost feature selection strictly inside training folds, (5) align dispersion and reconstruction parameterization with the official likelihood path, and (6) replace best-test-seed selection with nested CV or fixed external holdout evaluation.
